## 1. 개요

### 1.1 문제 정의

현업 사용자가 정보계 DB에 직접 접근하고 있어 데이터 유출·보안 통제가 불가하며,
접근 권한이 수작업으로 관리되어 퇴사/부서이동 시 권한 회수가 누락되는 구조적 문제가 있다.

금융권 감사 대응 차원에서도 SQL 실행 이력, 권한 신청/승인 이력, 개인정보 조회 이력을
위변조 불가한 형태로 1년 이상 보관해야 하나, 현재 체계에서는 이를 충족하지 못한다.

### 1.2 제안 솔루션

웹 기반 SQL 에디터(웹쿼리)와 권한관리 어플리케이션을 통합 구축하여,
사용자는 반드시 웹을 통해서만 정보계 DB에 접근하도록 강제한다.

모든 접근은 **권한 신청 → 다단계 결재 승인 → 자동 권한 부여/회수** 플로우를 거치며,
쿼리 실행 시 개인정보 마스킹·건수 제한·조회이력 적재가 자동 적용된다.

## 2. 권한 계층 모델

시스템의 접근 제어는 5단계 계층 구조로 관리된다. 권한 신청·검증·회수 등 모든 권한 관련 기능은 이 계층 모델을 기반으로 동작한다.

```
DataSource (DB 서버/클러스터)
  └── Schema (데이터베이스/스키마)
        └── Table (테이블)
              └── Column (컬럼) — 마스킹/복호화 대상
                    └── Operation (명령어) — SELECT, SHOW만 허용
```

### 2.1 계층별 정의

| 레벨               | 대상                | 설명                                         | 권한 신청 단위               | 예시                             |
| ------------------ | ------------------- | -------------------------------------------- | ---------------------------- | -------------------------------- |
| **L1. DataSource** | DB 서버/클러스터    | 접속 가능한 DB 서버 선택. 관리자가 사전 등록 | 접근 신청 (P0)               | `정보계-Oracle`, `분석계-Trino`  |
| **L2. Schema**     | 데이터베이스/스키마 | DataSource 내 스키마 선택                    | DataSource 접근 시 함께 지정 | `hdmf_prod`, `hdmf_mart`         |
| **L3. Table**      | 테이블              | 스키마 내 개별 테이블 SELECT 권한            | 테이블 단위 신청 (P0)        | `T_CUSTOMER`, `T_CONTRACT`       |
| **L4. Column**     | 컬럼 (개인정보)     | 마스킹 대상 컬럼의 복호화(원문 조회) 권한    | 컬럼 단위 신청 (P1)          | `주민번호`, `계좌번호` 원문 조회 |
| **L5. Operation**  | SQL 명령어          | 허용 명령어 제어 (MVP: SELECT/SHOW만)        | 시스템 정책 (신청 불필요)    | SELECT, SHOW                     |

### 2.2 권한 신청 플로우 (계층 기반)

```
[사용자]
  1. DataSource 선택 (접근 가능한 DB 서버 목록에서)
  2. Schema 선택 (해당 서버 내 스키마 목록에서)
  3. Table 검색 및 선택 (한글명/영문명 검색)
  4. 사유·기간 입력
  5. 결재 신청
     ↓
[결재 승인 후]
  → 해당 DataSource > Schema > Table에 대한 SELECT 권한 자동 부여
```

### 2.3 웹쿼리 에디터 연동

- SQL 에디터에서 DataSource/Schema를 먼저 선택해야 쿼리 실행 가능
- 자동완성은 선택된 DataSource > Schema 내 보유 권한 테이블만 노출
- 쿼리 실행 시 서버 측에서 DataSource > Schema > Table 3단계 권한 검증

### 2.4 관리자 DataSource 관리

| 기능                          | 설명                                               |
| ----------------------------- | -------------------------------------------------- |
| DataSource 등록/수정/삭제     | DB 서버 연결 정보 관리 (호스트, 포트, 드라이버 등) |
| Schema 동기화                 | DataSource 연결 후 스키마 목록 자동 수집           |
| 테이블/컬럼 메타데이터 동기화 | 스키마 내 테이블·컬럼 목록 주기적 동기화           |
| 마스킹 정책 설정              | 컬럼 단위 개인정보 마스킹 대상 지정                |

## 4. UX 시나리오 가이드

### 4.1 사용자 여정 맵 (End-to-End)

신규 사용자가 처음 로그인해서 데이터를 조회하기까지의 전체 여정이다.

```
Phase 1: 탐색              Phase 2: 신청            Phase 3: 대기            Phase 4: 실행
───────────────           ──────────────          ────────────────        ─────────────────

SQL 에디터 진입            권한 신청 화면             내 신청 목록              SQL 에디터
     │                        │                      │                      │
     ▼                        ▼                      ▼                      ▼
DS/Schema 선택           DS > Schema > Table       결재 진행 상태 확인        자동완성에 테이블 노출
     │                   사유·기간 입력              (1차→2차→최종)                │
     ▼                        │                      │                      ▼
SQL 작성 (자유롭게)             ▼                      ▼                  쿼리 실행 → 결과
     │                   결재 제출                 승인 완료 알림
     ▼                                           (이메일/메신저)
실행 시도
     │
     ├─ 권한 있음 → 결과 반환
     └─ 권한 없음 → 에러 + [권한 신청] 유도
```

### 4.2 핵심 UX 원칙: "SQL 먼저, 권한은 나중에"

권한이 없는 사용자도 SQL 에디터에 자유롭게 진입하고 쿼리를 작성할 수 있다. 실행 시점에 권한을 검증하고, 부족한 권한에 대해 바로 신청할 수 있도록 유도한다.

**이유:**

- 사용자는 대부분 "어떤 테이블이 필요한지" 이미 알고 있음
- SQL을 먼저 작성 → 부족한 권한만 신청하는 것이 자연스러운 워크플로우
- P1의 "쿼리 입력 시 테이블 자동 추출 신청"과 자연스럽게 연결됨

### 4.3 화면별 권한 상태 대응

| 화면                           | 권한 없을 때                                             | 권한 있을 때                            |
| ------------------------------ | -------------------------------------------------------- | --------------------------------------- |
| **SQL 에디터**                 | SQL 작성 가능, 실행 시 권한 없는 테이블 안내 + 신청 링크 | 정상 실행 + 결과 표시                   |
| **테이블 탐색기 (사이드패널)** | 전체 테이블 목록 표시, 권한 없는 테이블은 🔒 표시        | 권한 있는 테이블 🔓 강조, 자동완성 대상 |
| **권한 신청**                  | DS > Schema > Table 드릴다운 신청                        | (동일) 추가 테이블 신청                 |
| **내 신청 목록**               | 결재 진행 상태 실시간 확인                               | 승인 완료 이력 확인                     |

### 4.4 SQL 에디터 — 권한 없는 테이블 실행 시 UX

```
┌─────────────────────────────────────────────────────────────┐
│  SQL 에디터                                                  │
├────────────┬────────────────────────────────────────────────┤
│            │ DataSource: [정보계-Oracle ▾]  Schema: [hdmf ▾] │
│ 테이블 탐색기 ├────────────────────────────────────────────────┤
│            │                                                │
│ 🔓T_DEPT   │  SELECT c.name, c.dept                         │
│ 🔓T_CODE   │  FROM T_CUSTOMER c                             │
│ 🔒T_CUSTOMER│  JOIN T_CONTRACT ct ON c.id = ct.cust_id      │
│ 🔒T_CONTRACT│                                               │
│ 🔒T_ACCOUNT│                              [▶ 실행]          │
│            ├────────────────────────────────────────────────┤
│            │ ⚠️ 권한 부족 — 아래 테이블에 대한 접근 권한이 필요합니다  │
│ ────────── │                                                │
│ [권한 신청→]│  T_CUSTOMER  🔒 권한 없음  [신청]                 │
│            │  T_CONTRACT  🔒 권한 없음  [신청]                 │
│            │                                                │
│            │  [2개 테이블 일괄 권한 신청 →]                      │
└────────────┴────────────────────────────────────────────────┘
```

**동작 방식:**

1. 사용자가 SQL 작성 후 [실행] 클릭
2. 서버가 FROM/JOIN 절 파싱 → 권한 없는 테이블 목록 반환
3. 결과 패널에 부족 테이블 목록 + 개별 [신청] 버튼 + [일괄 신청] 버튼 노출
4. [신청] 클릭 시 권한 신청 화면으로 이동 (DataSource/Schema/Table 자동 입력)

### 4.5 SQL 에디터 — 보유 권한이 전혀 없는 경우 (첫 진입)

```
┌────────────┬────────────────────────────────────────────────┐
│            │ DataSource: [선택하세요 ▾]  Schema: [— ▾]       │
│ 테이블 탐색기 ├────────────────────────────────────────────────┤
│            │                                                │
│ (비어있음)  │  ┌──────────────────────────────────────┐      │
│            │  │                                      │      │
│            │  │  📋 조회 가능한 테이블이 없습니다        │      │
│            │  │                                      │      │
│            │  │  데이터를 조회하려면 먼저               │      │
│            │  │  테이블 접근 권한을 신청해주세요.        │      │
│            │  │                                      │      │
│            │  │  [권한 신청하러 가기 →]                 │      │
│            │  │                                      │      │
│            │  └──────────────────────────────────────┘      │
└────────────┴────────────────────────────────────────────────┘
```

**DataSource 선택 후 권한이 없는 경우:**

- 테이블 탐색기에 해당 Schema의 전체 테이블 목록은 표시하되 모두 🔒 상태
- 에디터는 정상 사용 가능 (SQL 작성 → 실행 시 권한 부족 안내)

## 6. 기술 명세

### 6.1 아키텍처 개요

```
[사용자 PC]
    │ HTTPS(443)만 허용
    ▼
[WEB — React + TypeScript / Nginx]
    │ REST API
    ▼
[WAS — Java / Spring Boot]
    ├── 권한 검증 레이어 (자체 구현)
    ├── 쿼리 실행 중계 (Trino JDBC or 직접 JDBC — TBD)
    ├── 마스킹 처리 (자체 구현)
    └── 이력 적재
    │
    ├──▶ [어플리케이션 DB] — 사용자 정보, 권한 이력, 결재 이력, 로그
    ├──▶ [SQL 엔진 — Trino or JDBC] ──▶ [정보계 DB]
    ├──▶ [HR/IAM 시스템] — 인사 연동 (일별 배치 + 이벤트)
    └──▶ [알림 서비스] — 이메일 / 사내 메신저
```

| 레이어          | 구성                                                          | 비고                                    |
| --------------- | ------------------------------------------------------------- | --------------------------------------- |
| 사용자 접근     | 내부망 PC → HTTPS(443)만 허용                                 | 사용자PC → Web 서버만 접근 가능         |
| WEB (Frontend)  | React + TypeScript, Nginx                                     | SPA, 내부망 배포                        |
| WAS (Backend)   | Java / Spring Boot                                            | 권한 검증, 쿼리 실행 중계, 이력 적재    |
| 어플리케이션 DB | 시스템 관리용 별도 DB                                         | 사용자 정보, 권한 이력, 결재 이력, 로그 |
| SQL 엔진        | Trino 또는 직접 JDBC 연결 (TBD)                               | 정보계 DB 쿼리 실행 전담                |
| 권한 제어 엔진  | **자체 구현** — DataSource/Schema/Table/Column 계층 접근 제어 | Spring Security 기반 정책 레이어        |
| 인사 연동       | IAM/SSO, HR 시스템 API                                        | 일별 동기화 + 실시간 이벤트             |
| 알림            | 이메일 / 사내 메신저 연동                                     | 결재 요청·완료·반려 알림                |

### 6.2 자체 구축 개발 범위

시스템 방향은 **자체 완전 구축**으로 확정. 외부 솔루션(QueryPie 등) 없이 전 기능을 직접 개발한다.

| 영역               | 자체 개발 항목                                                                       |
| ------------------ | ------------------------------------------------------------------------------------ |
| DataSource 관리    | DataSource 등록/연결 관리, Schema·테이블·컬럼 메타데이터 동기화                      |
| 권한관리           | 권한 신청/승인/회수 플로우, DataSource > Schema > Table > Column 계층 접근 제어 정책 |
| 결재 워크플로우    | 다단계 결재선 자동 생성, 전결/대결, 일괄 승인, 결재 이력 아카이빙                    |
| 웹쿼리 UI          | SQL 에디터, 결과 그리드, 마스킹 표시, 조회 사유 입력 팝업                            |
| 마스킹 처리        | 컬럼별 마스킹 정책 관리, 서버 응답 단계 마스킹 적용, 복호화 권한 처리                |
| SQL 실행 엔진 연동 | Trino JDBC 또는 직접 JDBC 연결 (TBD), DML 차단 필터                                  |
| 인사 연동          | HR/IAM API 연동 배치 + 이벤트, 권한 자동 부여/회수                                   |
| 감사 로그          | SQL 실행 이력, 결재 이력, 접근 로그 append-only 저장                                 |
| 관리자 화면        | 권한 현황, 이상행위 모니터링, 감사 자료 추출                                         |

### 6.6 보안 / 데이터 처리

**인증 및 접근 제어**

- 모든 통신 TLS/SSL 적용 (HTTPS only)
- 사용자PC → Web 서버 HTTPS(443) 접근만 허용, 직접 DB 접근 차단
- IAM/SSO 연동 인증, JWT 기반 세션 관리
- 비활성 idle timeout 자동 로그아웃 (정책값 적용)
- IP 대역 제한: 10.x.x.x 내부망만 허용

**쿼리 보안**

- 모든 쿼리 서버 측 파싱 후 DML 차단 (SELECT, SHOW만 허용)
- 권한 검증은 반드시 서버 측 처리 — 프론트 단독 검증 불가
- 개인정보 컬럼 마스킹은 서버 응답 단계에서 처리 (DevTools 노출 방지)
- SQL Injection 방지: 파라미터 바인딩 또는 쿼리 파싱 화이트리스트

**데이터 보관**

- SQL 실행 이력: append-only 구조, 수정/삭제 불가, 1년 이상 보관
- 결재 이력: IP, 타임스탬프, 결재자 의견 포함, 위변조 방지
- 개인정보 조회 사유: 사유 기입 후 이력 보관
